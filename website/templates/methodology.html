Methodology

Languages:

Everything here is written in python, stored in a MYSQL database, and displayed with django.

Individual Backtests:

Each backtest is done by taking OHLC price data for a specific asset (as a dataframe) and applying a strategy algorithm which generates of a list of . That list of trades is then used to generate a list of evaluation metrics (link) which are used to gauge the performance of the backtest. Some pieces of a strategy may use high/low prices as triggers or to generate some evaluation metrics. However, as things are now, all entries/exits are based on closing prices. Loss exits are recorded as the first closing price that breaks outside risk range. Profit exits are recorded as the first closing price that breaks profit targets (for strategies use profit targets) or the first closing price after an exit signal (for strategies that don't use profit targets). I understand using close prices may hurt performance (as some trades may have crazy stop loss slippage or may not take profit exits that would have otherwise been triggered by high/low prices) however I feel this serves as an adequate handicap to make returns more realistic. Additionally I did write something to account for slippage, commission, and interest rate carry, although this is not implemented in the results.

Parameter Permutations:

Creating permutations of parameters is based on the concept of taking a strategy and viewing its performance given a spectrum of input parameters. To get this spectrum, given X number of required parameters, a range of values is assigned to each parameter and a list of permutations is generated. The strategy is then run on selected price data for each item in that list of permutations.

For example if strategy X takes input parameters A,B and C, then a range of values is assigned to each parameter. So:

  • A gets [30,31,32 ... 100]
  • B gets [10,11,12 ... 30]
  • C gets [5,6,7 ... 15]

Then a list of permutations, or unique combinations of parameters inputs, is generated which looks something like:

  • {A:30,B:10,C:5}
  • {A:31, B:10, C:5}
  • {A:32, B:10, C:5}
  • ...
  • {A:100, B:30, C:15}

Strategy X is then run on selected price data for each item in that list of parameter inputs.

Obviously this method of brute force testing has its limits. First and foremost being limitations on the number of permutations that can be tested in a timely manner. Depending on the number of parameters as well as the length of the range each parameter is assigned, the total number of permutations can reach well into the hundreds of millions. Even after optimizing the code for speed and setting it up to run in parallel via multiprocessing, python is still python and python is slow.

To account for this, parameter ranges are spaced out in increments, i.e if the range is [1,2,3,4...100] the new range will be [1,5,10,15...100]. This vastly cuts down on the number of permutations while still yielding enough of a spectrum of parameter inputs to provide a good insight into whats working and what isn't.

Timeframes and Assets:

To cut down on the total number of backtests and reduce the run time needed to update the results, all strategies are run on a 1 hour timeframe (i.e 1 candle represents 1 hour). Backtests are run over a 1 month of time (for both the current month and previous month). The number of assets is also limited to a handful of tickers from forex, crypto and equity indices.
